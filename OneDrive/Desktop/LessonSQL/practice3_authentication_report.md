# Практика 3: Настройка различных методов аутентификации в PostgreSQL

Раздел: **Управление доступом в PostgreSQL**  
Необходимые знания: **Лекция 3: Аутентификация в PostgreSQL**

---

## 1) Цель работы

Настроить и протестировать методы аутентификации PostgreSQL (`trust`, `md5`, `scram-sha-256`, `peer`), выполнить миграцию с `md5` на `scram-sha-256`, включить логирование подключений и проанализировать записи логов.

---

## 2) Выполненные задания

## Часть 1. Подготовка окружения

### 1.1 Поиск конфигурационных файлов

Выполнены команды:

```sql
SHOW hba_file;
SHOW config_file;
SHOW data_directory;

SELECT name, setting
FROM pg_settings
WHERE name IN ('hba_file', 'config_file', 'data_directory');
```

Получены пути к:
- `pg_hba.conf`
- `postgresql.conf`
- каталогу данных (`data_directory`)

Сделана резервная копия `pg_hba.conf` перед изменениями.

### 1.2 Создание тестовых пользователей

Созданы пользователи:
- `user_scram` (с `scram-sha-256`)
- `user_md5` (с `md5`)
- `user_alice` (с `scram-sha-256`)

Проверка типа хеша через `pg_shadow` показала разные префиксы:
- `md5...`
- `SCRAM-SHA-256$...`

Почему `scram-sha-256` безопаснее `md5`:
- использует современную схему challenge-response;
- устойчивее к оффлайн-подбору;
- не опирается на устаревший `md5`, известный коллизиями;
- лучше подходит для production и современных стандартов безопасности.

## Часть 2. Настройка методов аутентификации

### 2.1 Метод `trust` (только тестовая среда)

В `pg_hba.conf` временно добавлялись правила:

```conf
host    all   user_scram   127.0.0.1/32   trust
host    all   user_scram   ::1/128        trust
```

Результат:
- `user_scram` подключается без пароля (успешно);
- другие пользователи без `trust`-правил пароль требуют;
- после теста `trust` удален как небезопасный.

### 2.2 Метод `md5`

Добавлены правила:

```conf
host    all   user_md5     127.0.0.1/32   md5
host    all   user_md5     ::1/128        md5
```

Результат:
- правильный пароль: подключение успешно;
- неправильный пароль: отказ аутентификации.

### 2.3 Метод `scram-sha-256`

Настроено:

```sql
ALTER SYSTEM SET password_encryption = 'scram-sha-256';
SELECT pg_reload_conf();
ALTER USER user_alice WITH PASSWORD 'NewSecurePassword789!';
```

В `pg_hba.conf`:

```conf
host    all   user_alice   127.0.0.1/32   scram-sha-256
host    all   user_alice   ::1/128        scram-sha-256
```

Проверено:
- `user_alice` успешно проходит аутентификацию;
- `user_md5` с `scram`-правилом получает ошибку (ожидаемо, пока не мигрирован).

### 2.4 Метод `peer`

На Windows выполнен теоретически (ограничение платформы).  
Для Linux/Unix пример:

```conf
local   all   user_alice   peer
```

Смысл: вход в PostgreSQL разрешается только при совпадении пользователя ОС и роли БД.

## Часть 3. Миграция с md5 на scram-sha-256

### 3.1 Аудит пользователей

Выполнен аудит:

```sql
SELECT usename, ... FROM pg_shadow;
```

Выявлены пользователи с `md5`, требующие обновления.

### 3.2 Обновление паролей

Выполнено:

```sql
ALTER USER user_md5 WITH PASSWORD 'NewSecurePassword123!';
```

После изменения хеш стал `SCRAM-SHA-256...`.  
Правила `pg_hba.conf` обновлены с `md5` на `scram-sha-256`.

### 3.3 Финальная безопасная конфигурация `pg_hba.conf`

```conf
# TYPE   DATABASE  USER      ADDRESS           METHOD
local    all       postgres                     peer
local    all       all                          scram-sha-256
host     all       all       127.0.0.1/32      scram-sha-256
host     all       all       ::1/128           scram-sha-256
host     all       all       192.168.1.0/24    scram-sha-256
host     all       all       0.0.0.0/0         reject
```

Пояснение: порядок строк критичен, применяется первое совпавшее правило.

## Часть 4. Аудит и логирование

### 4.1 Включение логирования подключений

Применены параметры:

```sql
ALTER SYSTEM SET log_connections = on;
ALTER SYSTEM SET log_disconnections = on;
ALTER SYSTEM SET log_line_prefix = '%t [%p]: [%l-1] user=%u,db=%d,app=%a,client=%h ';
SELECT pg_reload_conf();
```

Проверены:
- `SHOW log_connections;`
- `SHOW log_disconnections;`
- `SHOW log_line_prefix;`

### 4.2 Анализ неудачных попыток входа

Смоделированы 3–5 неудачных логинов (неверный пароль/метод).  
В логах обнаружены записи с:
- временем события;
- пользователем;
- клиентским IP;
- БД;
- причиной отказа.

Использование логов для безопасности:
- выявление brute-force и перебора паролей;
- обнаружение аномальной географии IP;
- контроль несанкционированных попыток доступа;
- база для SIEM/алертов.

## 3) Таблица результатов тестирования

| Метод | Пользователь | Условия | Результат |
|---|---|---|---|
| trust | user_scram | localhost, trust-правило включено | Успех без пароля |
| trust | user_md5 | localhost, trust только для user_scram | Пароль требуется / отказ без пароля |
| md5 | user_md5 | верный пароль | Успех |
| md5 | user_md5 | неверный пароль | Ошибка аутентификации |
| scram-sha-256 | user_alice | правило scram + новый пароль | Успех |
| scram-sha-256 | user_md5 (до миграции) | scram-правило, md5-хеш | Ошибка |
| scram-sha-256 | user_md5 (после миграции) | scram-правило, новый пароль | Успех |
| peer | user_alice | Linux/Unix, совпадение пользователя ОС | Успех |
| peer | user_alice | Linux/Unix, другой пользователь ОС | Ошибка |

## 4) Анализ безопасности методов

- `trust`: допустим только для временных локальных тестов, в production запрещен.
- `md5`: устаревший, поддерживать только на этапе миграции.
- `scram-sha-256`: рекомендуемый метод для production.
- `peer`: удобен для локального администрирования на Unix-системах.
- `cert` (SSL): повышает доверие к клиенту через сертификаты, рекомендуется для критичных сред.

## 5) Ответы на контрольные вопросы

1. **Аутентификация vs авторизация**: аутентификация проверяет личность, авторизация определяет права.  
2. **Почему SCRAM лучше MD5**: современный и более устойчивый механизм проверки секрета.  
3. **Порядок правил в `pg_hba.conf`**: сверху вниз, действует первое совпадение.  
4. **Когда допустим trust**: только в изолированной тестовой среде.  
5. **Как работает peer**: PostgreSQL доверяет пользователю ОС при локальном подключении (Unix).  
6. **Плюсы SSL/cert**: защита канала + подтверждение клиента сертификатом.  
7. **Влияние миграции md5 -> scram**: старые md5-хеши и md5-правила нужно обновить, иначе вход может ломаться.  
8. **Что видно в логах аутентификации**: время, IP, пользователь, БД, статус входа, причина ошибки.
